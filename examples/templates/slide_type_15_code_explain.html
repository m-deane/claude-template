from sklearn.pipeline import Pipeline
from sklearn.compose import ColumnTransformer
from sklearn.impute import SimpleImputer
from sklearn.preprocessing import StandardScaler
from sklearn.ensemble import GradientBoostingClassifier

# Define feature groups
numeric_features = ['age', 'income', 'tenure']
categorical_features = ['plan_type', 'region']

# Build preprocessing pipeline
preprocessor = ColumnTransformer([
    ('num', Pipeline([
        ('imputer', SimpleImputer()),
        ('scaler', StandardScaler()),
    ]), numeric_features),
    ('cat', OneHotEncoder(
        handle_unknown='ignore'
    ), categorical_features),
])

# Full pipeline: preprocess → train
model = Pipeline([
    ('prep', preprocessor),
    ('clf', GradientBoostingClassifier(
        n_estimators=200,
        learning_rate=0.1,
    )),
])

Feature Groups

Separate numeric and categorical columns so each gets the right preprocessing treatment.

Column Transformer

Routes each column group through its own sub-pipeline. Numeric columns get imputed and scaled; categorical columns get one-hot encoded.

End-to-End Pipeline

Chains preprocessing and model training into a single object that prevents data leakage and simplifies deployment.

This slide walks through our sklearn pipeline architecture. On the left is the actual code. On the right, three key design decisions. First, we separate numeric and categorical features upfront so each gets appropriate preprocessing. Second, the ColumnTransformer lets us apply different transformations to different column groups in a single step. Third, wrapping everything in a Pipeline object means we can call fit and predict once, with no risk of data leakage between train and test sets. This pattern is the foundation of every model we deploy to production.

This three-step progression shows our end-to-end pipeline. Step one: we use Polars with lazy evaluation to scan 12 million rows of parquet files from S3, filtering server-side for just the data we need. This takes 3.2 seconds compared to 45 seconds with pandas. Step two: feature engineering using Polars' expressive column API — rolling averages, categorical encoding, and derived ratios in a single chained operation. Step three: we feed the features into a LightGBM classifier. The key insight here is that each step is independently testable and the entire pipeline runs in under 30 seconds for 12 million rows.
