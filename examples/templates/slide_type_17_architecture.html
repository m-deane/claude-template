This diagram shows our complete ML pipeline from data sources to monitoring. Data arrives from S3 and Kafka into our Spark Streaming ingestion layer. From there, dbt and Airflow handle transformation and orchestration. The feature store — which I highlighted because it was the biggest architectural win — uses Feast to ensure consistent features between training and serving. XGBoost handles the modeling, served via FastAPI with sub-25-millisecond latency. Finally, Grafana monitors prediction drift and system health. The entire pipeline from raw event to production prediction runs in under 90 seconds.

This vertical stack shows how we organized our ML system into six layers, each with clear responsibilities. Starting from the bottom: Infrastructure runs on Kubernetes with dedicated GPU clusters managed via Terraform. The data platform uses Snowflake for batch and Kafka for streaming. Feature engineering sits on top with Feast and dbt. Training and experimentation use MLflow for tracking and Optuna for hyperparameter optimization. Model deployment uses KServe with built-in A/B testing. And monitoring at the top uses Evidently AI for drift detection alongside standard Prometheus/Grafana. The key insight is that each layer can scale and evolve independently — our infrastructure team can upgrade the GPU cluster without touching the feature engineering layer, and the ML team can experiment with new models without involving DevOps.

This hub-and-spoke diagram shows how our ML Platform sits at the center, providing shared capabilities to six domain-specific services. The Data Lake feeds training data into the platform. The Fraud detection team uses real-time inference for transaction scoring. NLP handles customer support ticket classification using Transformers. Analytics consumes model outputs through Looker dashboards. The Recommendation System uses embedding-based similarity from the platform. And Computer Vision uses YOLO v8 for product image classification. The ML Platform provides shared infrastructure — feature storage, model registry, experiment tracking, and serving — so that each domain team focuses on their specific use case rather than rebuilding ML infrastructure from scratch. This architecture lets us support six use cases with a platform team of four engineers.
